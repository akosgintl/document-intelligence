# Review is a caller-owned API resolution, not a managed review service

Introducing Review-actor semantics raised whether this system should run review itself (an internal reviewer team/UI, à la AWS A2I) or treat Review purely as an API surface the caller's own side resolves. We chose the latter: the caller submits a resolution — a Document Type for `classification_needs_review`, or a full field-value set (validated against the bound Schema exactly like automated Extraction) for `extraction_needs_review`/`extraction_failed` — via API; this system never runs its own reviewer team or UI. This keeps Document Intelligence a pure classification/extraction API product rather than adding a review-ops business, a much larger commitment — reinforced by AWS deprecating A2I (their equivalent managed-review product) to new customers this month. Consequences: no Reviewer identity or audit trail is modeled (resolution is a stateless status transition, not a persisted Review record); a classification resolution can only assign/confirm a Document Type for the Document as already split (bad page-boundary splits aren't Review-fixable in v1); and a failed extraction resolution goes straight to `extraction_failed` with no retry, since there's no Model Provider round-trip to retry against.

## Considered Options

- **Managed review-ops** (rejected): build an internal reviewer team/UI/queue like AWS A2I. Rejected as scope creep into a different business, reinforced by A2I's closure to new customers.
- **Patch-style extraction resolution** (rejected): let the caller submit corrections only for low-confidence fields, leaving the rest untouched. Rejected because it needs a second, partial-validation code path and is ambiguous under cross-field Schema constraints; full replacement reuses the exact validation path automated Extraction already has.
