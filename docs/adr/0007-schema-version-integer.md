# Schema version identifiers are plain incrementing integers, not semver

Nothing in the pipeline consumes compatibility semantics (major/minor/patch) between Schema versions — "latest version" lookups and version binding (e.g. Review's resolution to a Type's latest Schema version, ADR-0003) only need a total, unambiguous order. We chose a plain incrementing integer, scoped per Document Type, over semver: semver would ask the Schema-authoring operator to correctly classify every change's compatibility impact — a judgment call teams get wrong even with practice — for a signal nothing in this system currently reads. The version lives in the Schema Registry's file/directory naming convention, not as a field inside the Schema JSON content itself, keeping the Schema artifact a clean, standard JSON Schema document rather than one decorated with registry-specific bookkeeping.
