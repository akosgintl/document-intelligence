# Provider-exhausted Documents route into the existing needs_review statuses, not a new status

A Document whose Classification or Extraction call exhausts the transient-error retry budget (three attempts, per the retry/backoff policy) is a failure mode this domain model hadn't named — distinct from a worker crash (Job-level `failed`, ADR-0005) and distinct from a genuine low-confidence or invalid-content result. We chose to route it into the existing `classification_needs_review`/`extraction_needs_review` statuses rather than invent a new one: Review's resolution contract (ADR-0003) already covers "assign a Type or confirm `unclassified`" / "submit a full field replacement" regardless of *why* automated processing didn't finish, so reusing it costs nothing new — no new status, no new Review mechanics. A single Document's Provider exhaustion also doesn't escalate to Job-level `failed`, consistent with ADR-0002/ADR-0005's separation of Document-level outcomes from pipeline-level ones: other Documents in the same Submission may process fine even if one hit a stubborn rate limit.
