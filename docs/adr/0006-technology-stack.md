# Technology stack: FastAPI + PostgreSQL + arq/Redis + S3-compatible storage

The PRD names its infrastructure needs generically ("a relational database," "a job queue library and broker") without picking concrete technologies. We picked a single, cohesive async-first stack: **Python** (already implicit from the repo's initial setup) with **FastAPI** for the HTTP API layer, matching the PRD's async-processing and global-concurrency-limiting requirements (stories 6, 26, 27) with native `async`/`await` rather than a bolted-on async story. **PostgreSQL** for persistence, chosen for native `JSONB` support over extracted-Field payloads, keeping Job/Document results genuinely queryable per the PRD's own goal (story 36) rather than opaque blobs. **`arq`** (not Celery) as the task queue against Redis: worker tasks calling the Model Provider are pure I/O-bound async work, and `arq`'s tasks are plain `async def` functions, so the same `asyncio.Semaphore`-based concurrency limiter used at the API layer works unchanged inside worker tasks — Celery's synchronous-per-worker execution model would need bridging (`asyncio.run()` or `gevent`/`eventlet` monkey-patching) to do the same job. **Object storage** via an S3-compatible client (`aioboto3`), backed by MinIO as a plain container in local Docker Compose and any real S3-compatible bucket in production — one code path for both, avoiding a local-filesystem branch that would diverge from production behavior (violating the PRD's own object-storage rationale, story 32).

## Considered Options

- **Celery + Redis/RabbitMQ** (rejected): the mature, default Python task-queue choice, but its synchronous-per-worker execution model fights the async-native design chosen for the API layer; its broader feature set (periodic tasks via Celery Beat, complex routing) isn't needed here.
- **Django** (rejected): brings a full ORM/admin/templating ecosystem this project doesn't need — no user-facing web UI, single-tenant, nothing that plays to Django's strengths.
- **Local filesystem for dev object storage** (rejected): reintroduces the dev/prod divergence the PRD's object-storage requirement (story 32) exists specifically to avoid.
