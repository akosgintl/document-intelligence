# Job lifecycle gains a `failed` status, distinct from Document-level failures

The PRD's original Job lifecycle (`pending` → `processing` → `complete`) had no way to represent the pipeline itself failing to finish a Submission — e.g. a worker crashing repeatedly — leaving a caller unable to distinguish "still working" from "will never finish," which undercuts the PRD's own goal that Job status should be trustworthy. We added a `failed` terminal Job status, reached only after bounded infrastructure-level recovery attempts are exhausted (exact retry mechanics depend on the chosen queue broker, still open on the map). `failed` is orthogonal to Document-level failure statuses (`extraction_failed`, etc.): those never block `complete`, per ADR-0002, because they describe one Document's content, while `failed` specifically means the pipeline couldn't finish the Submission at all. Documents that completed before the failure remain visible in the Job's result rather than being discarded, consistent with this system's existing page-accounting invariant against silently dropping work.
