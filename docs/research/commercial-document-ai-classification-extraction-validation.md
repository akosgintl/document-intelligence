# Commercial document-intelligence platforms: Classification, Extraction, Validation

Research date: 2026-07-10.

## Scope and method

This note surveys how four established commercial document-intelligence platforms — AWS, Microsoft, Google, and LlamaCloud (LlamaIndex) — approach the three tasks this repo implements: **Classification** (assigning a Document Type), **Extraction** (pulling structured fields per a Schema), and **Validation** (including cross-field rules and confidence/fuzzy scoring). Every claim below is cited to a primary vendor documentation page that was actually fetched or directly quoted from an official-domain search result during this research session (docs.aws.amazon.com, learn.microsoft.com, docs.cloud.google.com/cloud.google.com, docs.cloud.llamaindex.ai/developers.llamaindex.ai). No secondary blog posts, listicles, or comparison articles were used. Where a specific field name or mechanism could not be confirmed from a fetched primary source, this is stated explicitly rather than guessed.

This repo's own vocabulary (Submission, Document, Document Type, Schema, Page, Classification, Extraction, Schema Registry, Model Provider, Job) is defined in `CONTEXT.md`; the two-phase page-level/document-level classification design is defined in `docs/adr/0001-two-phase-classification.md`. Both are referenced throughout, but neither is modified by this file.

## 1. AWS — Textract, Bedrock Data Automation, Comprehend, Augmented AI

### Classification

AWS splits "classification" across three separate services rather than treating it as one capability of a single product:

- **Amazon Comprehend custom classification** trains a model in either **multi-class** mode (one class per document) or **multi-label** mode (one or more classes per document) from labeled example documents ["Custom classification"](https://docs.aws.amazon.com/comprehend/latest/dg/how-document-classification.html). The real-time `ClassifyDocument` API's response syntax returns:
  ```json
  {
     "Classes": [ { "Name": "string", "Page": number, "Score": number } ],
     "DocumentMetadata": { ... },
     "DocumentType": [ { "Page": number, "Type": "string" } ],
     "Errors": [ ... ],
     "Labels": [ { "Name": "string", "Page": number, "Score": number } ],
     "Warnings": [ ... ]
  }
  ```
  — `Classes` for multi-class mode (mutually-exclusive), `Labels` for multi-label mode (non-exclusive categories), each entry carrying a `Score` in the 0–1 range verbatim from the [`ClassifyDocument` API reference](https://docs.aws.amazon.com/comprehend/latest/APIReference/API_ClassifyDocument.html). Asynchronous batch classification jobs emit one JSON line per input line/file, e.g. `{"File": "file1.txt", "Line": "0", "Classes": [{"Name": "Documentary", "Score": 0.8642}, ...]}`, per ["Inputs and outputs for asynchronous classification jobs"](https://docs.aws.amazon.com/comprehend/latest/dg/how-class-run.html).
- **Amazon Bedrock Data Automation (BDA)** classifies via **blueprints**: "you can classify documents by assigning a document class and providing a description when you create a blueprint. The document class serves as a high-level categorization of the document type" — each page is classified into a document type and matched to the corresponding blueprint before extraction runs, per ["How Bedrock Data Automation works"](https://docs.aws.amazon.com/bedrock/latest/userguide/bda-how-it-works.html) and ["Create Blueprints for Classification"](https://docs.aws.amazon.com/bedrock/latest/userguide/idp-cases-classification.html).
- **Amazon Textract** has no general-purpose classification API; its **Queries** feature (below) answers targeted questions rather than assigning a document type, and its `AnalyzeExpense`/`AnalyzeID` are fixed, vendor-defined document-type parsers, not a trainable Document Type registry.

### Extraction

- **Textract `AnalyzeExpense`** returns an `ExpenseDocument` containing `LineItemGroups` (tables, parsed into `LineItems`) and `SummaryFields` (everything outside a table), each an array of `ExpenseField` objects. Verbatim `ExpenseField` shape, per the [`ExpenseField` API reference](https://docs.aws.amazon.com/textract/latest/dg/API_ExpenseField.html):
  ```
  ExpenseField { Currency, GroupProperties, LabelDetection: ExpenseDetection,
                 PageNumber, Type: ExpenseType, ValueDetection: ExpenseDetection }
  ```
  Both `LabelDetection` and `ValueDetection` are `ExpenseDetection` objects — each independently carries its own confidence and geometry, i.e. the *label* and the *value* are confidence-scored separately, not just the value ([`ExpenseDocument` reference](https://docs.aws.amazon.com/textract/latest/dg/API_ExpenseDocument.html)).
- **Textract Queries** is Textract's ad hoc extraction mechanism: a caller submits a `Query { Text, Alias, Pages }` (e.g. `Text: "What is the patient first name?"`, `Alias: "PATIENT_FIRST_NAME"`) per the [`Query` API reference](https://docs.aws.amazon.com/textract/latest/dg/API_Query.html), and gets back a `QUERY` block linked via an `ANSWER` relationship to a `QUERY_RESULT` block. Verbatim example from ["Queries"](https://docs.aws.amazon.com/textract/latest/dg/queryresponse.html):
  ```json
  { "BlockType": "QUERY", "Query": { "Text": "What is the patient first name?", "Alias": "PATIENT_FIRST_NAME" },
    "Relationships": [{ "Type": "ANSWER", "Ids": ["21396475-..."] }] },
  { "BlockType": "QUERY_RESULT", "Confidence": 1.0, "Text": "ALEJANDRO", "Id": "21396475-..." }
  ```
  Queries has no notion of a registered Schema/Document Type at all — every extraction is a free-form question, evaluated independently, with its own confidence.
- **BDA blueprints** are the closest AWS analog to this repo's Schema: a blueprint is "a list of field names that you can extract, the data format ... such as string, number, or boolean—as well as natural language context for each field that you can use to specify data normalization and validation rules," per ["Blueprints"](https://docs.aws.amazon.com/bedrock/latest/userguide/bda-blueprint-info.html). Each field carries: `Field name`, `Description` (a natural-language prompt doubling as normalization/validation instructions, e.g. `"Is the year of birth before 1992?"`), `Type` (string/number/boolean/array of string/array of numbers), `Confidence score`, `Extraction Type` (explicit vs. inferred), and `Page Number`. Blueprints also support `Table Fields`, `Groups`, and `Custom types` for nested structures. Notably: **"Audio and Image blueprints do not return a confidence score"** — confidence is not universal even within one AWS product.

### Validation

- No dedicated cross-field validation rule language was found in any fetched AWS documentation (Textract, Comprehend, or BDA). BDA's blueprint field `Description` can encode validation-like instructions in natural language (the "before 1992" example above), but this is a per-field prompt fed to the underlying model, not a declarative expression language evaluated against the full extracted result.
- **Amazon Augmented AI (A2I)** is AWS's explicit, named human-review mechanism. Core components, per ["Core Components of Amazon A2I"](https://docs.aws.amazon.com/sagemaker/latest/dg/a2i-getting-started-core-components.html): a **human review workflow** (a.k.a. **flow definition**) bundles a **work team**, a **worker task template**, and (for built-in task types) **activation conditions**; when those conditions are met, a **human loop** is created and tasks are routed to the work team. For the built-in Textract task type, "Amazon A2I monitors the Amazon Textract results and sends a document to one or more human workers for review when the conditions specified in your flow definition are met" — and those conditions are explicitly confidence-threshold based on Textract's key-value pair confidence scores.
- **Deprecation note (directly relevant, timely):** the A2I documentation states as of this research: **"After careful consideration, we have made the decision to close new customer access to AWS A2I, effective 7/30/26. Existing customers can continue to use the service as normal... we do not plan to introduce new features."** Given today's date (2026-07-10), AWS's managed human-review-loop product is closing to new customers in roughly three weeks. This is a live, notable data point (see synthesis).

## 2. Microsoft — Azure AI Document Intelligence (+ Content Understanding)

### Classification

- **Custom classification models** classify each page of an input file into one of several classes and can also identify document boundaries in a multi-document file. Training requires "as few as five training documents per document class" and at least two classes, with training input built from Layout-model output per document, per ["Document Intelligence custom classification model"](https://learn.microsoft.com/azure/ai-services/document-intelligence/train/custom-classifier?view=doc-intel-4.0.0) and ["Build and train a custom classification model"](https://learn.microsoft.com/azure/ai-services/document-intelligence/how-to-guides/build-a-custom-classifier?view=doc-intel-4.0.0). Test results surface "a confidence score for each document identified and the page range for each of the documents identified" — i.e. confidence is attached per identified document instance, not just per page.
- Azure explicitly documents a **classify-then-extract two-call pattern** as one alternative to a **composed model** (single call, service auto-selects the best-matching extraction model): "Call the classification model based on the document class. **This step allows for a confidence-based check before invoking the extraction model analysis.**" — per ["Document Intelligence custom classification model" capability table](https://learn.microsoft.com/azure/ai-services/document-intelligence/train/custom-classifier?view=doc-intel-4.0.0#model-capabilities). This is a documented instance of gating a second (extraction) model call on the confidence of a first (classification) call — directly comparable to, but more conditional than, this repo's ADR-0001 design, which always re-runs document-level classification unconditionally rather than gating it on page-level confidence.
- **Azure Content Understanding** (newer, generative-AI-based product, distinct from Document Intelligence) supports classification as one of three field **generation methods** (`extract`, `classify`, `generate`) and also as a document-level **segmentation/classification** step via `contentCategories` in the analyzer config, which can route mixed-type input to the right analyzer, per ["What is a Content Understanding analyzer?"](https://learn.microsoft.com/azure/ai-services/content-understanding/concepts/analyzer-reference).

### Extraction

- Azure's Schema equivalent is defined per model type. **Prebuilt models** (e.g. `prebuilt-invoice`) have Microsoft-maintained field schemas with fixed types — "InvoiceDate is returned as a date type and SubTotal as a currency type" — needing no configuration. **Custom models** require the caller to label fields and explicitly assign each a type (`string`, `number`, `integer`, `date`, plus `time`, `phoneNumber`, `currency`, `address` for prebuilt), which drives auto-normalization, per ["What is Azure Document Intelligence"](https://learn.microsoft.com/azure/ai-services/document-intelligence/overview?view=doc-intel-4.0.0#custom-models).
- Two custom extraction model families: **custom template** (visual-template-based, tolerant of little layout variation, supports key-value pairs/selection marks/tables/signatures/regions) and **custom neural** (deep-learning based, tolerant of layout variation across e.g. different vendors' W-2s, supports overlapping fields), per ["Document Intelligence custom models"](https://learn.microsoft.com/azure/ai-services/document-intelligence/train/custom-model?view=doc-intel-4.0.0). **Custom composed models** combine several standalone models behind one endpoint.
- **Content Understanding's `fieldSchema`** is the more direct analog to this repo's JSON-Schema-based Schema: a named schema with a `fields` object, where each field definition carries `type` (`string`/`number`/`boolean`/`date`/`object`/`array`), `description` (explicitly "the AI model processes this description as a mini-prompt to guide field extraction" — not just documentation), and `method` (`extract`/`classify`/`generate`), with `object`/`array` types supporting nested structures ([analyzer reference](https://learn.microsoft.com/azure/ai-services/content-understanding/concepts/analyzer-reference#field-configuration)). Confidence and source grounding are opt-in per field or analyzer-wide via `estimateFieldSourceAndConfidence` / `estimateSourceAndConfidence`.
- Extraction result JSON exposes bounding-region and span data throughout — `boundingRegions`, `spans`, `content` — on paragraphs, tables, key-value pairs, and fields, per ["Analyze document API response"](https://learn.microsoft.com/azure/ai-services/document-intelligence/concept/analyze-document-response?view=doc-intel-4.0.0) and the `DocumentFieldOutput`/`AnalyzedDocument` SDK reference pages.

### Validation

- **Confidence is explicitly layered**, not a single number. Per ["Interpret and improve accuracy and confidence scores"](https://learn.microsoft.com/azure/ai-services/document-intelligence/concept/accuracy-confidence?view=doc-intel-4.0.0): (1) **document type confidence score** (how closely the whole document matches training data), (2) **field level confidence** (per labeled field), (3) **word confidence score** (per OCR'd word, matched to a field via `spans`), (4) **selection mark confidence score**; as of the `2024-11-30 (GA)` API, table/row/cell confidence was added too. Azure's explicit guidance is to combine these into "a comprehensive confidence for the extracted result" rather than trust field confidence alone.
- **Confidence value semantics** are stated plainly: "a confidence value of 0.95 (95%) indicates that the prediction is likely correct 19 out of 20 times... confidence can be used to determine whether to automatically accept the prediction or flag it for human review."
- **A concrete straight-through-processing (STP) threshold example is documented**: "the customer might submit results with confidence values greater than or equal to .80 for straight-through processing and apply human review to results with confidence values less than .80," per the ["Transparency note and use cases for Document Intelligence"](https://learn.microsoft.com/azure/foundry/responsible-ai/document-intelligence/transparency-note#evaluation-of-document-intelligence). The same page names **"Human-in-the-loop"** as an explicit, numbered Responsible-AI principle: "Keep a human in the loop... Ensure that you can have real-time human intervention in the solution to prevent harm. A human in the loop enables you to manage situations when Document Intelligence does not perform as required."
- **Content Understanding** frames the same idea as "Confidence scoring: Automate with control" — auto-approve above a threshold, route below-threshold fields to a human, with a worked example (billing address at confidence 0.96 posted automatically; amount due at confidence 0.52 routed to a human), per ["Validate document analyzer quality with confidence, grounding, and labeled samples"](https://learn.microsoft.com/azure/ai-services/content-understanding/document/analyzer-improvement#confidence-scoring-automate-with-control"). That same article frames a three-part QA loop: **confidence scoring** (automate/flag), **grounding** (traceability to source), and **labeled samples** ("in-context learning" — adding a corrected example to improve future extractions of that pattern).
- **No dedicated cross-field/business-rule validation expression language was found** in any fetched Azure Document Intelligence or Content Understanding documentation (no equivalent of "total = sum of line items" as a declarative rule) — this is a documented absence relative to Google's CEL mechanism below, not a guess.

## 3. Google — Document AI

### Classification

- **Custom Document Classifier** processors use a flat, user-defined label set (Google's docs state plainly: "Labels are flat and don't support child entities"). Each label can carry an optional description that functions as a classification prompt to help disambiguate similar categories, per ["Create, use, and manage a custom document classifier"](https://docs.cloud.google.com/document-ai/docs/custom-classifier). Training data requirements: "a minimum of one document in both the training and test sets for each document type," with "at least 10 documents for each label" recommended for best performance. **Confidence scores are supported for custom classifier models in Preview, and for best performance, they should be used with fine-tuned models** (same source) — i.e. Google flags confidence-score support as not-yet-fully-general for this processor type.
- Classification results are returned in the same `entities[]` array used for extraction: each entity's `type` field holds the predicted label (empty string if unclassified) and `confidence` holds the score, per the ["Process a document using a Splitter/Classifier processor"](https://docs.cloud.google.com/document-ai/docs/samples/documentai-process-splitter-document) sample. Google's **Splitter/Classifier** processor performs page-level document-boundary-finding and classification together in one processor call — structurally similar in intent to (but mechanically different from) this repo's page-level classification pass described in ADR-0001, since Google does not describe a second, document-level re-classification pass to settle a single authoritative type/schema version the way this repo's ADR-0001 does.

### Extraction

- **Custom Extractor** processors offer three modeling options, trading off training-data volume against tolerance for layout variation, per ["Custom extractor overview"](https://docs.cloud.google.com/document-ai/docs/custom-extractor-overview): **Foundation model** (generative AI, handles high-to-low layout variation, ~0–50+ documents), **Custom model** (model-based, low-to-medium layout variation, 10–100+ documents), and **Template-based** (fixed layout only, as few as 3 documents).
- Schema fields are defined with a **Data type**, an **Occurrence**, and a **Description** ("additional context, insights, and prior knowledge for each entity to improve extraction accuracy"). Occurrence values found: `OPTIONAL_ONCE`, `REQUIRED_ONCE`, `OPTIONAL_MULTIPLE`, `REQUIRED_MULTIPLE`, per ["Custom extractor mechanisms"](https://docs.cloud.google.com/document-ai/docs/ce-mechanisms) — this is Google's closest equivalent to JSON Schema's `required` plus array cardinality, expressed as a single enum rather than two orthogonal JSON Schema keywords. Custom extractor also supports **derived fields** ("populated through intelligent inference or generation based on document context, rather than direct text extraction") and a dedicated **signature** entity type "derived using visual cues."
- Extraction result shape, verbatim from ["Handle processing response"](https://docs.cloud.google.com/document-ai/docs/handle-response):
  ```json
  {
    "textAnchor": { "textSegments": [{"startIndex": "14", "endIndex": "24"}], "content": "2020/01/01" },
    "type": "invoice_date",
    "confidence": 0.9938466,
    "pageAnchor": {},
    "id": "2",
    "normalizedValue": { "text": "2020-01-01", "dateValue": {"year": 2020, "month": 1, "day": 1} }
  }
  ```
  Nested/child entities reuse this exact shape inside a `properties[]` array. Common normalized types include `dateTime`, `currency`, `money`, and `number`.

### Validation

- Google is the only one of the four platforms surveyed with a **named, general-purpose, documented cross-field validation mechanism**: the **CEL (Common Expression Language) dialect for document validation**, per ["CEL dialect for document validation"](https://docs.cloud.google.com/document-ai/docs/ce-cel-validation). Expressions evaluate against a root variable `doc` exposing each entity's `mention_text`, `normalized_value`, and `bounding_poly`. Documented verbatim examples include exactly the kinds of cross-field rules this repo's task description calls out:
  - Date ordering: `doc.due_date.normalized_value > doc.invoice_date.normalized_value`
  - Sum reconciliation: `doc.line_items.map(item, item.total.normalized_value).sum() == 275.0` (with epsilon-based floating-point tolerance, ±0.01, so this isn't brittle exact-equality)
  - Structural parity: `doc.employer_contribution.size() == doc.employee_deduction.size()`
  It also supports spatial/layout validation via `checkVerticalAlignment`/`checkHorizontalAlignment` functions, string-only restrictions on `contains()`/`startsWith()`, and standard CEL list/math libraries plus a `reduce()` macro. Rules return a boolean and are configured as part of the extraction schema/processor, not bolted on downstream.
- **Human-in-the-Loop (HITL)** is Google's named managed human-review product: confidence-based **validation filters** (configurable at document level or per-label) determine which documents get queued for human review by a **specialist pool**, per the [HITL quickstart](https://docs.cloud.google.com/document-ai/docs/hitl/quickstart). **Deprecation note:** Google's own release notes state HITL "was deprecated on January 16, 2024" ([HITL release notes](https://docs.cloud.google.com/document-ai/docs/hitl/whats-new)); Google's current guidance is to work with a certified partner (e.g. Devoteam, Searce, Quantiphi) for human review/correction instead of using Google's own product. This is the second of two major vendors in this survey whose managed, first-party human-review-loop product is being sunset (AWS A2I closing to new customers 7/30/26; Google HITL already deprecated) even as the underlying confidence-threshold *pattern* remains standard.

## 4. LlamaCloud (LlamaIndex) — LlamaExtract and Classify

### Classification

LlamaIndex's classification product is **Classify**, part of the LlamaParse platform family and distinct from LlamaExtract. Per the [Classify getting-started guide](https://developers.llamaindex.ai/python/cloud/llamaclassify/getting_started/) and search-grounded excerpts of the same page family:

- A taxonomy is a list of **rules**; each rule is `{ type, description }` — `type` is the label (constrained to alphanumeric characters, spaces, hyphens, underscores; examples given: `"invoice"`, `"sec_filing"`, `"Tax Return"`, `"10-K"`), and `description` is "a natural-language description of the content that should match this type."
- Two operating modes: **Fast** (text-only classification, cheaper, good when content rather than layout determines type) and **Multimodal** (vision-model-based, for visually rich documents).
- Per-file result fields: **`type`** (predicted label), **`confidence`** (0.0–1.0), and **`reasoning`** (a step-by-step natural-language explanation of the classification decision) — reasoning attached directly to the classification result is distinctive versus AWS/Azure/Google, none of which return a natural-language rationale alongside the label+score.
- **`matching_threshold`** (range 0.1–0.99, default 0.6) is a request-level parameter: the minimum confidence required for a match to be considered acceptable. Unlike AWS/Azure/Google, where the confidence threshold lives in a *separate* human-review product configuration, LlamaCloud's threshold is a first-class parameter of the classification call itself.

### Extraction

LlamaExtract's **Data Schema** is this repo's closest commercial analog, being literally either **JSON Schema or a Pydantic model** — the same base mechanism this repo already uses for its Schema, per ["LlamaExtract Schema Design"](https://docs.cloud.llamaindex.ai/llamaextract/features/schema_design) and [Core Concepts](https://developers.llamaindex.ai/python/cloud/llamaextract/features/concepts). Design guidance given: keep schema nesting to 3–4 levels, make fields optional when data may be absent, use arrays for variable-count entities, and — matching Azure Content Understanding's pattern — "use descriptions to pass formatting instructions or few-shot examples," since "descriptions function as additional context for the underlying LLM," not merely documentation.

- **Extraction Target** controls granularity, and is LlamaExtract's answer to something like this repo's Page/Document split, though structured differently: `per_doc` (default — one schema instance for the whole document), `per_page` (schema applied independently per page, array of results), `per_table_row` (schema applied per row/entity in a detected table), per the [options reference](https://developers.llamaindex.ai/python/cloud/llamaextract/features/options).
- **Extraction Config** parameters: `data_schema`, extraction/parse **tier** (`cost_effective` / `agentic` [documented as the recommended default] / `agentic_plus` [upcoming, for "complex, high-stakes extractions"]), `cite_sources` (bool), `confidence_scores` (bool), `system_prompt`, `target_pages`, `max_pages`, and `version` pinning (a `YYYY-MM-DD` date or `"latest"`, for reproducibility as the underlying models/prompts change over time).
- An **Extraction Job** is the asynchronous task; an **Extraction Run** is its result: the schema-shaped extracted data plus, when enabled, **citations** (per field: `page` number and `matching_text` — the verbatim source text supporting the value) and **reasoning** ("a brief explanation for the extracted value based on the text provided **or an error message in case the text does not contain enough information to extract the field**" — i.e. an explicit signal for "field not supportable" distinct from a low-but-present confidence score), per ["Extract Data from Financial Reports - with Citations and Reasoning"](https://developers.llamaindex.ai/python/cloud/llamaextract/examples/extract_data_with_citations). Usage metadata (`num_pages_extracted`, `num_document_tokens`, `num_output_tokens`) is also returned.

### Validation

- No dedicated cross-field/business-rule validation expression language (comparable to Google's CEL) was found in any fetched LlamaExtract documentation. The two validation-adjacent mechanisms documented are: (1) the underlying **JSON Schema/Pydantic** type constraints on the Data Schema itself (structural validation only — the same limitation this repo's prior research file already established for JSON Schema in general), and (2) the opt-in **`confidence_scores`** and **`cite_sources`**/reasoning outputs, which surface trust signals per field but leave the accept/reject threshold and any human-review workflow entirely to the caller. **No built-in human-review queue, work-team, or HITL product analogous to AWS A2I, Azure's Studio review loop, or Google's (now-deprecated) HITL was found for LlamaCloud** — Classify's `matching_threshold` is the only place a confidence gate is built into the product itself, and even that only rejects/accepts a match rather than routing it anywhere for review.

## Cross-cutting synthesis

### What recurs across 3 or more platforms

1. **Confidence score as a first-class field on every classification and extraction result.** Every platform surveyed attaches a numeric confidence (0–1 or a percentage) to individual classification labels and individual extracted field values: AWS (`Score` on Comprehend `Classes`/`Labels`, `Confidence` on Textract Query results and `ExpenseDetection`, BDA's per-field "Confidence score"), Azure (`confidence` at document-type, field, word, selection-mark, and table/row/cell granularity), Google (`confidence` on every `entity`, classification included), and LlamaCloud (`confidence` on Classify results, opt-in `confidence_scores` on LlamaExtract fields). This is the single most universal pattern found.
2. **Confidence-threshold-gated human review as the standard fallback pattern**, named explicitly by three of four vendors: AWS Augmented AI (A2I), Azure's documented STP threshold (≥0.80 auto, <0.80 human) plus its explicit "Human-in-the-loop" Responsible-AI principle, and Google's Human-in-the-Loop (HITL) validation filters. LlamaCloud is the partial exception: it exposes the confidence signal and even a request-level `matching_threshold` on Classify, but does not ship a human-review queue/workflow product — that piece is left to the integrator.
3. **Schema-driven extraction, expressed as a named field list with type + natural-language description per field, is near-universal**: AWS BDA blueprints, Azure Content Understanding's `fieldSchema`, Google's custom extractor schema (type/occurrence/description), and LlamaExtract's JSON-Schema/Pydantic Data Schema all converge on the same shape — field name, type, and a description that is explicitly documented (by Azure, AWS BDA, and LlamaCloud alike) to double as a prompt/instruction to the underlying model, not just human-facing documentation. This repo's Schema is already JSON Schema with a `description` per field used "both as classification guidance and as the extraction target" (per `CONTEXT.md`), so this repo's design already aligns with the majority pattern here.
4. **Field-level source grounding (geometry/page/span/citation) is standard**, though named differently everywhere: AWS `Geometry`, Azure `boundingRegions`/`spans`, Google `textAnchor`/`pageAnchor`, LlamaCloud `citations` (`page` + `matching_text`). This repo's domain model has no equivalent concept at all — extraction results are presumably plain structured field values with no notion of "where in the Document this came from."
5. **Two-step classify-then-extract pipelines, with the second step sometimes explicitly gated on the first step's confidence.** Azure documents this outright ("This step allows for a confidence-based check before invoking the extraction model analysis"); Google's Splitter/Classifier processor performs page grouping and classification together ahead of a separate extraction processor call. This repo's ADR-0001 two-phase design (page-level classification for boundary-finding, then a full document-level re-classification to settle one authoritative Schema version) is a *third* variant: unlike Azure's confidence-gated second call, this repo's document-level classification pass always runs unconditionally — it is not gated on the page-level pass's confidence, because (per ADR-0001) this repo currently has no page-level confidence signal to gate on in the first place.

### What's platform-specific

- **Google's CEL dialect for document validation is the only documented, general-purpose, declarative cross-field validation mechanism found across all four platforms.** Neither AWS, Azure, nor LlamaCloud has an equivalent — AWS's closest analog is a natural-language instruction embedded in a BDA blueprint field description (a prompt, not a rule engine); Azure and LlamaCloud rely entirely on confidence scores plus (for Azure) human review, or (for LlamaCloud) per-field reasoning/citations, to catch what a cross-field rule would otherwise catch deterministically.
- **LlamaCloud's `reasoning` field** (a natural-language explanation attached to both classification and extraction results, including an explicit "insufficient information" error case for extraction) is distinctive — no other platform surveyed returns a rationale string alongside the label/value and confidence number.
- **Managed human-review-loop products from the two largest vendors are both being sunset right now**: AWS A2I is closing to new customers effective 2026-07-30 (per its own documentation, and given today's date this is roughly three weeks away), and Google Document AI HITL was deprecated 2024-01-16. This doesn't mean the underlying pattern (confidence threshold → human review) is going away — it remains universal per point 2 above — but it suggests vendors are moving away from offering a turnkey reviewer-workforce product themselves and toward exposing the confidence signal for integrators to build their own review surface on top of, which is exactly the posture LlamaCloud already has natively.
- **This repo's retry-with-validation-errors-fed-back-to-the-Provider pattern** (per `CONTEXT.md`'s Extraction definition: one retry on validation failure, with the validation errors fed back to the Model Provider before falling to `extraction_failed`) was not found documented as an automated, per-request mechanism on any of the four platforms surveyed. Azure's closest analog — "labeled samples (in-context learning)" — is a manual, offline correction loop (add a corrected example, the analyzer "generalizes better" on future documents of that pattern), not a live retry within the same extraction call. This is a point where this repo's design is *more* sophisticated than what's documented for any platform surveyed, not less.

### Where this repo's current design diverges from common commercial practice — stated plainly

Per `CONTEXT.md` and ADR-0001, this repo's domain vocabulary — Submission, Document, Page, Document Type, Schema, Classification, Extraction, Schema Registry, Model Provider, Job — **has no confidence score concept anywhere**, and **no human-in-the-loop concept anywhere**. Every one of the four commercial platforms surveyed treats a numeric confidence value as a first-class, always-present (or at minimum commonly-available) field on both classification and extraction results, and three of the four (AWS, Microsoft, Google) treat a confidence-threshold-gated human/manual review step as a standard, explicitly named architectural pattern — in Azure's words, a way "to manage situations when [the service] does not perform as required."

This repo's Document lifecycle — `pending` → `classified`/`unclassified` (terminal) → `extracted`/`extraction_failed` (terminal) — has no state representing "the model produced a plausible but low-confidence result, pending review." A Document is either accepted outright (classified, or successfully extracted against its bound Schema version) or it fails outright (`unclassified`, or `extraction_failed` after one retry). There is no middle ground for the exact scenario every surveyed platform's confidence score exists to capture: a result the model is unsure about but did not outright fail to produce. Similarly, this repo's Validation is JSON-Schema structural validation only (confirmed by the prior research file at `docs/research/json-schema-for-classification-extraction-validation.md`), with no cross-field/business-rule mechanism comparable to Google's CEL dialect, and no source-grounding (bounding box/page/citation) concept comparable to what all four platforms attach to extracted values.

None of this is inherently wrong — it may be a deliberate simplification given this repo's current scope, and the retry-with-errors-fed-back mechanism arguably substitutes automated self-correction for some of what a human-review step would otherwise catch. But given how consistently confidence scoring and a human-review fallback recur across AWS, Microsoft, and Google specifically, this is a gap worth a conscious decision rather than a silent omission if/when this repo's design is revisited.
